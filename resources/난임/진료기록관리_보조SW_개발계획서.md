# 진료기록 관리 보조 소프트웨어 개발 계획서

> **문서 성격**: 시스템 개발 계획서 (초안)
> **대상 시스템(가칭)**: 난임 진료기록 관리 보조 소프트웨어
> **작성 기준일**: 2026-06-21
> **법적 면책**: 본 문서는 규제 구조에 대한 분석에 기반한 개발 계획이며 법률 자문이 아닙니다. 실제 적용 전 개인정보·의료법 전문가의 검토를 권고합니다.

## 1. 개요

### 1.1 목적
난임 환자 진료기록의 **작성·관리·출력을 보조**하는 데스크톱 응용 소프트웨어를 개발한다. 본 시스템은 의료인이 진료 내용을 구조화하여 입력하고, 이를 규정에 맞는 서식으로 출력하도록 지원한다.

### 1.2 시스템의 법적 성격 (핵심 전제)
본 시스템은 **엄밀한 의미의 전자의무기록(EMR) 시스템이 아니다.** 다음 전제 위에서 설계·운영된다.

- **공식 진료기록부 = 출력 후 자필서명한 종이 문서** (의료법 제22조의 종이 진료기록부)
- **소프트웨어 및 내부 DB = 진료기록 작성을 돕는 보조 사본** (진본이 아님)
- 진료기록의 전자적 작성·보관(의료법 제23조 전자의무기록)을 **선택하지 않는다.** 따라서 전자서명 시스템, 전자의무기록 시설·장비 고시, 외부보관 강화기준, EMR 인증제 적용 대상에서 벗어난다.

> 의료법 제23조는 "전자문서로 작성·보관할 수 **있다**"는 임의규정이므로, 종이 진료기록부를 진본으로 두는 본 모델은 적법하게 성립한다. 다만 이 성립은 **"종이를 진본으로 일관되게 운영"한다는 전제가 실제로 지켜질 때만** 유효하다(§3.3, §6 참조).

### 1.3 적용 범위 및 가정
| 항목 | 가정 |
|---|---|
| 운영 주체 | 단일 의료기관(난임 진료) |
| 배포 형태 | 로컬 설치형 데스크톱 앱 (의료기관 내부) |
| 진본 매체 | 출력·서명된 종이 |
| 데이터 외부 전송 | 없음(완전 로컬) — 클라우드/외부보관 미사용 |
| 청구 연동 | 1차 범위 제외(추후 검토) |

## 2. 규제 요건 분석

### 2.1 의료법 제22조 — 종이 진료기록부가 진본
- 의료인은 진료기록부를 갖추어 두고 의료행위에 관한 사항과 의견을 **상세히 기록하고 서명**하여야 한다.
- 필수 기재사항(의료법 시행규칙 제14조 제1항 제1호)을 **누락 없이** 포함해야 한다(§부록 A).
- 부실 기재 시 형사처벌(의료법 제90조, 300만원 이하 벌금) 및 면허자격정지 처분 대상이 될 수 있다.
- 진료기록부는 **한글로 기록하도록 노력**해야 한다(시행규칙 제14조 제2항).
- 보존기간: 진료기록부 **10년**(시행규칙 제15조). → 종이 보존 의무.

### 2.2 의료법 제23조 회피 — 무엇을 면제받는가
종이를 진본으로 두므로 다음 요건의 직접 적용을 받지 않는다.

- 전자서명법에 따른 전자서명 기재 의무
- 「전자의무기록의 관리·보존에 필요한 시설과 장비에 관한 기준」(보건복지부고시) 충족 의무
- 외부(클라우드) 보관 시 강화기준·위탁계획서 제출·정기점검
- 전자의무기록시스템 인증제

### 2.3 개인정보보호법 — 그대로 적용되는 의무 (회피 불가)
본 시스템의 DB에는 **실제 난임 환자의 민감정보(건강정보)** 와 경우에 따라 **고유식별정보(주민등록번호)** 가 저장된다. 따라서 보조도구여도 **개인정보처리시스템**으로서 「개인정보의 안전성 확보조치 기준」(개인정보보호위원회 고시)을 준수해야 한다.

| 의무 | 본 시스템 적용 |
|---|---|
| 저장 시 암호화 | 주민등록번호는 암호화 보관 의무. 민감정보·고유식별정보 안전성 확보조치 대상 → **DB at-rest 암호화 필수** |
| 접근통제 | 계정·권한 관리, 최소권한, 무단접근 통제 |
| 접속기록 보관 | 일반 1년 이상 / **민감정보·고유식별정보 처리 시스템은 2년 이상**. 위·변조 방지, 별도 보관, 정기 백업 |
| 내부관리계획 | 수립·시행 |
| 백업 | 분실·훼손 대비(내부) |
| 유출 대응 | 유출 시 통지·신고 의무, 손해배상 책임 |

> **핵심 결론**: 의료법 제23조는 피하더라도 개인정보보호법은 피할 수 없다. 따라서 **암호화 + 접근통제 + 2년치 위변조 방지 감사로그**는 EMR을 만들 때와 사실상 동일하게 구현해야 한다.

### 2.4 요건 요약
- **반드시 구현**: at-rest 암호화, 접근통제, 감사로그(2년·위변조 방지·별도 보관), 백업, 필수기재항목 출력
- **면제(미적용)**: 전자서명, 시설·장비 고시, 외부보관 강화기준, EMR 인증
- **운영으로 보장**: 종이 진본 일관성, 지체 없는 출력·서명, 정정·보존 동선

## 4. 기술 스택 검토 및 선정

### 4.1 선정 기준
1. 민감정보 at-rest 암호화 용이성 및 보안 표면 최소화
2. 고품질 인쇄/서식 출력(한국 의료 현장은 인쇄 의존도 높음)
3. 위변조 방지 감사로그 구현 용이성
4. 개발 생산성(기존 React/Expo 역량 재사용 가능 여부)
5. 단일 단말 → 다단말 확장 경로

### 4.2 Tauri + SQLite/SQLCipher 평가
**적합도: 단일 의료인·단일 단말 환경에서 사실상 최적에 가까움.**

| 장점 | 단점/유의 |
|---|---|
| 경량(작은 바이너리), Rust 코어 → 보안 표면이 Electron보다 작음 → 민감정보 처리에 유리 | 생태계가 Electron/.NET보다 작고, 백엔드 커스터마이즈에 Rust 학습 필요 |
| 웹 프론트엔드(React) 사용 → 기존 역량 재사용 | SQLite는 **단일 writer** 구조 → 다단말 동시쓰기에 부적합 |
| SQLCipher로 투명한 AES-256 at-rest 암호화 | 키 관리는 별도 설계 필요(아래) |
| 크로스플랫폼(Win/Mac/Linux) | 모바일은 베타(본 용도엔 불필요) |

### 4.3 대안 비교
| 스택 | 강점 | 약점 | 본 용도 적합성 |
|---|---|---|---|
| **Tauri + SQLCipher SQLite** | 경량·보안 표면 최소·React 재사용 | SQLite 동시성, Rust 러닝커브 | ★★★★☆ (단일 단말 최적) |
| **.NET (WPF/WinUI 3) + EF Core + SQLCipher** | Windows 친화(국내 의료 환경 다수), 인쇄·리포팅 성숙, DPAPI 키관리 용이 | Windows 종속, 웹 역량 재사용 불가 | ★★★★☆ (Windows 전용 프로덕션에 강함) |
| **Electron + better-sqlite3 + SQLCipher** | 생태계·채용 용이 | 무겁고 보안 표면 큼 | ★★★☆☆ |
| **Flutter desktop + SQLCipher** | UI 우수, 크로스플랫폼 | 데스크톱 성숙도·기존 역량 미스매치 | ★★★☆☆ |
| **로컬 온프레미스 서버(Go/Node/Rust + PostgreSQL) + Thin client** | 다단말 동시성·중앙 감사로그·백업 일원화 | 구축·운영 복잡도↑ | ★★★★☆ (다단말 필수 시 최적) |

### 4.4 시나리오별 권고
- **1인 의사·1대 단말(현재 가정)** → **Tauri + SQLCipher SQLite 유지 권고.** 가장 경량·안전하고 기존 React 역량을 살릴 수 있다. 현재 요건에는 이 조합이 최선에 가깝다.
- **Windows 전용으로 프로덕션 안정성·인쇄 품질을 최우선**으로 한다면 → **.NET(WinUI 3)** 가 국내 의료 현장 적합성에서 강력한 대안. 정부 서식/인쇄/HWP 연계가 많아질수록 유리.
- **한 기관 내 다단말(진료실+간호+원무)로 확장 예정** → SQLite의 단일 writer 한계가 드러나므로 **의료기관 내부에 소형 온프레미스 서버 + PostgreSQL**로 이행하고 단말은 Thin client(Tauri 또는 브라우저)로 둔다. 서버가 기관 내부에 있으므로 여전히 "내부 보관"이며 외부보관 강화기준을 피한다.

> **결론**: "Tauri + SQLCipher가 무조건 최선"이라기보다, **단일 단말 1차 버전의 최적해**다. 다단말 확장 가능성이 있으면 데이터 계층을 PostgreSQL로 갈아탈 수 있도록 **DB 추상화 계층을 처음부터 두는 것**을 권고한다.

### 4.5 암호화 키 관리 (스택과 무관한 핵심 난제)
- SQLCipher 키를 **하드코딩하거나 평문 파일로 저장하지 않는다.**
- OS 키스토어 사용: Windows DPAPI/Credential Manager, macOS Keychain.
- 기기 분실 시 복호화가 불가능하도록 키를 사용자 인증(로그인)과 결합.
- 백업본도 동일 정책으로 암호화하고, 키와 백업을 분리 보관.

### 4.6 감사로그 무결성 설계
- `audit_log` 테이블을 **append-only**로 운영(수정·삭제 트리거 차단).
- 각 레코드에 직전 레코드 해시를 포함하는 **해시 체인**으로 사후 변조 탐지.
- 주기적(예: 일/주 단위)으로 감사로그를 **별도 물리 저장장치**에 내보내 보관(안전성 확보조치 기준의 "별도 보관·정기 백업" 충족).
- 보관기간 **2년 이상**(민감정보 처리 시스템 기준).

## 5. 시스템 아키텍처

### 5.1 구성 개요
```
[ Tauri Desktop App ]
  ├─ Frontend (React)        : 입력 UI, 서식 미리보기, 출력
  ├─ Core (Rust)             : 비즈니스 로직, 키관리, 감사로그
  ├─ SQLCipher (SQLite)      : 암호화 로컬 DB (보조 사본)
  ├─ Print/PDF Renderer      : 진료기록부 서식 출력
  └─ Backup/Export Module    : 내부 백업 + 감사로그 별도 내보내기
[ OS Keystore ]              : 암호화 키 보호
[ 종이 진료기록부철 ]        : ★ 공식 진본 (10년 보존)
```

### 5.2 데이터 모델 개요(요약)
- `patient` : 인적사항(주민번호 등 민감 필드는 암호화 컬럼)
- `encounter` : 진료 건(진료일시, 주된 증상, 진단명, 진료경과, 치료내용)
- `infertility_detail` : 난임 특화(주기·시술·약물 등)
- `printout` : 출력본 스냅샷(생성일시·사용자·문서 해시·서명상태)
- `audit_log` : append-only 해시체인 감사로그
- `user` / `role` : 계정·권한

### 5.3 출력·서명 워크플로우
1. 의료인 진료 내용 입력
2. 필수기재항목 검증(누락 시 출력 차단/경고)
3. 진료기록부 서식 출력(PDF/인쇄) → `printout` 스냅샷·해시 저장
4. **출력물에 자필서명**
5. 서명된 종이를 진료기록부철에 **지체 없이** 보존
6. 모든 단계 `audit_log` 기록

## 6. 의료기관(의료인) 운영 안내

> 이 시스템의 적법성은 **운영 준수**에 결정적으로 의존한다. 아래 내용을 운영 매뉴얼 및 내부관리계획에 반영한다.

### 6.1 핵심 원칙
- **공식 진료기록부는 출력·서명된 종이입니다. 프로그램 화면·DB는 보조 자료입니다.**
- 진료 직후 **지체 없이** 출력하여 자필서명하고 보존합니다.

### 6.2 일상 운영 절차
1. 로그인(개인 계정) 후 진료 내용 입력
2. 필수기재항목 확인 → 출력
3. 출력물에 자필서명
4. 서명본을 진료기록부철에 편철·보존(10년)
5. 단말 이석 시 화면 잠금

### 6.3 정정·수정 절차
- 서명·보존 완료된 종이 원본은 **폐기하지 않습니다.**
- 정정이 필요하면 정정 내용을 별도로 출력·서명하여 **원본과 함께** 보존합니다(원본 + 정정본 모두 유지).
- 프로그램상 수정은 감사로그에 자동 기록됩니다.
- 환자는 본인 기록(추가기재·수정 시 **수정 전 원본 포함**) 열람·사본발급을 요청할 수 있으므로(의료법 제21조), 원본 보존이 특히 중요합니다.

### 6.4 보존 절차
- 종이 진료기록부 **10년 보존**(화재·수해·분실 대비 보관).
- 프로그램 데이터는 정기 백업(내부 매체), 백업본 암호화.

### 6.5 보안·계정 운영
- 계정은 개인별로 부여하고 공유하지 않습니다(접속기록 식별성 확보).
- 비밀번호 관리, 단말 물리 보안(잠금장치), 자동잠금 설정.
- 퇴직자 계정 즉시 말소.

### 6.6 내부관리계획 반영 사항
- 시스템의 보조도구 성격 및 종이 진본 원칙 명문화
- 접근권한 부여·변경·말소 절차
- 감사로그 보관(2년)·별도 매체 내보내기 주기
- 백업 주기·복구 절차
- 유출 사고 대응 절차(통지·신고)

## 8. 리스크 및 법적 한계
| 리스크 | 대응 |
|---|---|
| 이중기록으로 진본성 다툼 | 종이 진본 원칙 운영·UI 명시·출력 스냅샷 추적 |
| 개인정보 유출 | at-rest 암호화·접근통제·감사로그·키 분리 |
| 종이 보존 부담(10년) | 보존 동선·물리 보관 가이드 |
| 청구 연동 부재로 이중입력 | 1차 범위 외, 추후 연동 검토 |
| 다단말 확장 시 SQLite 한계 | DB 추상화 계층 선반영, PostgreSQL 이행 경로 확보 |

**한계**: 본 시스템은 EMR이 아니므로 전자적 진료기록의 법적 효력, 진료정보교류, 청구 연동 등 EMR의 이점은 제공하지 않는다. 또한 개인정보보호법상 의무(암호화·접속기록·유출책임)는 그대로 부담한다.
